# 申请书写作风格指南

本文件是申请书正文写作的唯一权威宪法。08-section-write Skill 和 grant-writer subagent 必须各自独立读取本文件。

---

## 1. 全局写作约束

### 1.1 写作前必须确认的信息

对每个 unit，动笔前必须确认：
1. 本 unit 在全文中的论证角色（来自 writing_units.yaml 的 role_in_document）
2. 本 unit 依赖前面哪些 unit 的结论（来自 outline_state.yaml 的 depends_on）
3. 本 unit 为后面哪些 unit 做铺垫（来自 outline_state.yaml 的 feeds_into）
4. 关键术语是否与已写好的 unit 一致
5. 本 unit 所属 section 在论证链中的位置
6. heading_output_rule：是否为所属 section 的第 1 个 unit

### 1.2 写作时的全局约束

**⚠️ Markdown 换行铁律（P0）**：pandoc 转换 markdown 到 docx 时，**紧邻的两行之间如果没有空行，转换后视为同一段落，不会换行**。因此，**所有段落之间、标题前后、参考文献条目之间、图片占位前后、表格前后，以及任何需要在 docx 中分行显示的单元之间，必须各保留一个空行**。这是所有写作和组装阶段的全局底层规则，违反此规则将导致 docx 输出内容全部挤在一起。

1. 术语首次出现原则：首次出现必须定义，后续 unit 沿用不重复解释
1.1 **英文缩写首次出现必须给出全称**：所有英文缩写（如 RDMA、QP、QoS、LLM、NVMe 等）在正文中首次出现时必须写出完整中文翻译+英文全称+（缩写），例如"远程直接内存访问（Remote Direct Memory Access, RDMA）"。同一缩写在本 unit 内后续出现直接用缩写即可，但跨 unit 时——如果该缩写在上游 unit 中已首次定义，本 unit 可沿用缩写不再展开
2. 承上启下：开头 1-2 句承接前文结论，结尾 1-2 句引出下文
3. 不越界：只写本 unit 负责的论证内容，不侵入同一 section 下其他 unit 的领地
4. 证据准确：引用论文数据时必须与 writing_units.yaml 的 sources 字段一致
5. 语气一致：与已写好的 unit 保持一致
6. 标题规则：所属 section 的第 1 个 unit 写 heading_level 级标题；后续 unit 以正文段落开头，不重复同级标题

### 1.3 写作后的自查

- 本 unit 结论是否与 depends_on 中 unit 的内容一致？
- 本 unit 的表述是否为 feeds_into 中 unit 留了正确的接口？
- 术语使用是否与已写好的 unit 一致？
- 所有 paragraph_slots 是否都已覆盖？

## 2. 职责边界（写作铁律）

**必须做：**
1. 严格按 unit 蓝图的 paragraph_slots / required_elements / core_argument / sources / avoid / writing_notes 写作
2. 判断 heading_output_rule 并正确输出标题层级
3. 将 unit 蓝图按 paragraph_slots 槽位扩写成正式申请书正文
4. 写入独立的 unit .md 文件

**不允许做：**
1. 不编造论文、数据、结论或技术指标
2. 不修改已写好的 unit
3. 不越过 unit 的 avoid 约束自行决定写什么
4. 不向正文中引入 unit 蓝图未提及的新核心论点或新创新点
5. 不跳过 paragraph_slots：每个 slot 必须在正文中有对应段落，或充分理由说明为何跳过
6. 不合并多个 unit .md 文件
7. 不执行全局审阅（属于 09-assemble）

## 3. 段落级写作规范

### 3.1 section_intro 类型的特殊处理

**heading-only（needs_intro_paragraph=false）**：
- 只输出一行 markdown 标题，不生成任何正文段落
- 不读取 L2/L3 上下文

**需要开篇段落（needs_intro_paragraph=true）**：
- 输出标题 + 1-3 段简短开篇（150-500 字）
- 开篇内容：本节在全书论证链中的角色（1 句）+ 本节要回答什么问题（1 句）+ 内容组织方式（1-2 句）
- 禁止：深入子级 section 技术细节、堆砌子级标题列表、泛泛的"本节将详细介绍……"

### 3.2 动笔前：基于 paragraph_slots 梳理段落

1. 通读所有 slots，确认理解每个 slot 的论证任务
2. 检查 slot 之间的逻辑递进关系
3. 对 must_include 中的每个要点确认有足够的上游知识支撑
4. 核心论点必须在正文中清晰体现

### 3.3 严禁脚手架泄漏（最重要的一条）

paragraph_slots 是内部施工图纸。slot 的 role 描述和 slot_id（P1, P2...）绝对不能穿透到正文输出中。

**禁令清单（以下所有形式均禁止出现在正文中）**：

| 禁止形式 | 示例 |
|---|---|
| slot_id 标签 | P1：...、P2： |
| slot role 做粗体段标题 | 路线一：本地多层存储优化。 |
| 编号小标题 | 1. 应用背景、（一）研究现状 |
| 编号代替研究内容 | 正文中用 M1-M4 指代四个模块、用 R1-R4 指代四项研究内容——始终用自然语言名称，不得用编号代替 |
| 清单式罗列 | 第一，...第二，...第三，...（超过一处即禁止） |
| 模板化过渡句 | 综上所述，...（每段结尾都用即禁止） |

**正确做法**：段落主题句自然推进。slot 的 role 是写作者的理解提示——理解后用**自己的语言**写成该段的主题句，不要让 role 文本原样出现在段落开头。

```
错误：路线一：本地多层存储优化。该路线的核心思想是在单节点内...
正确：在本地存储优化方面，现有工作通过在单节点内构建 GPU 显存、DRAM、NVMe SSD 的多层存储体系来压缩权重加载时间。

错误：上述工作的共同盲区。以上三条路线的加速类工作在各自场景下...
正确：然而，上述加速类工作存在一个关键的共同约束假设：它们均假定所利用的高速网络在当前时刻处于独占或近独占状态。

错误：路线二：硬件优先级队列隔离。DualPath 面向 agentic 推理场景...
正确：第二类工作是硬件级优先级队列隔离。DualPath 面向 agentic 推理场景...
（注意：这里"第二类工作"是自然语言过渡，不是粗体标签）
```

**核心判断标准**：写完一个 unit 后自问——"如果评审人读到这段，他能看出 paragraph_slots 的施工痕迹吗？"如果能，就是泄漏。

**正文内段落编号例外**：仅当 heading_level=5 时，正文段落可使用（1）（2）（3）作为段落内部分项编号，每节独立从（1）开始。这是正文叙述的一部分，不是 heading 编号。

### 3.4 关键技术展开规则

- 涉及核心方法、关键模块设计、独特技术路线选择时，必须用具体的技术描述：架构思路、关键设计选择及理由、典型工作流程或数据流
- 用例子和场景让技术可理解
- 图片留空机制：`> **[图 X：{标题}]** *{图片简要描述}*`。图片留空不能替代文字论证
- 表格自动生成机制：若 batch instruction 提供 `table_specs[]`，必须直接生成 Markdown 表格；不得把表格当作图片占位，也不得留下“表格待补”占位，除非关键数据缺失并在 manifest 中记录原因
- 引用 tag 机制：引用论文时使用 `{{cite:tag}}`，不得写 `[1]`、`[2]` 等数字编号；编号由 09-assemble 统一生成
- 禁止空洞套话："采用基于深度学习的特征提取方法""通过优化算法提升系统性能"
- 允许：给出具体模型、结构、选择理由和文献依据

## 4. 文风指南

### 4.1 语气

- 客观严谨：用事实和数据说话，避免主观臆断
- 自信但不自大：展示实力但不夸张，用成果说话
- 简洁有力：多用短句和动宾结构，避免冗长从句
- 逻辑清晰：每段有明确的论点，段落间有清晰的过渡

### 4.2 人称与时态

- 全文用"本项目"而非"我们"
- 立项依据用现在时/过去时，研究方案用将来时

### 4.3 常用动宾结构

揭示……机理、阐明……机制、建立……模型、发展……方法、实现……调控、突破……瓶颈、解决……难题、提出……策略

### 4.4 语气示例

好的语气：
> 已有研究表明……，但在……方面仍存在以下关键问题尚未解决：（1）……；（2）……。本项目拟从……角度出发，采用……方法，系统研究……，以期揭示……的内在机理。

不好的语气：
> 目前国内外对这个问题的研究还很少，我们将填补这一空白，在国际上首次实现……

### 4.5 严禁的套话和表述

| 类别 | 禁止内容 |
|------|---------|
| 夸大套话 | "填补空白""国内领先""国际先进""首次提出""世界首次" |
| 模糊表述 | "进行相关研究""开展一系列实验" |
| 口语化 | "做一下""看看""搞清楚" |
| 过度谦虚 | "本项目尝试性地探索……" |
| 过度自信 | "必将取得重大突破" |
| 无依据量化 | 不编造百分比、具体数字，除非来自 sources |
